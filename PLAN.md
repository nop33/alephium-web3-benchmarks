# Plan: Set Up Benchmarking Infrastructure + Update ROADMAP

## Context

We need to capture "before" metrics for the Alephium web3 monorepo modernization so we can demonstrate the impact of each roadmap phase. This involves two things:
1. Setting up benchmarking tools and apps that measure bundle size, tree-shaking effectiveness, packaging health, and cross-environment compatibility.
2. Updating ROADMAP.md to include a new "Phase 0: Benchmarking Baseline" section.

## Implementation Steps

### Step 1: Add size-limit to @alephium/web3

In `alephium-web3/packages/web3/package.json`:

- Add devDependencies: `size-limit`, `@size-limit/esbuild`, `@size-limit/file`
- Add `"size": "npx size-limit"` script
- Add `"size-limit"` config array with these scenarios:
  - Full import (`import *`)
  - Single function (`{ isValidAddress }`)
  - Codec namespace (`{ codec }`)
  - Utils namespace (`{ utils }`)
  - UMD bundle file size

Note: Since the package is CJS-only, all named import scenarios will pull the full module graph вЂ” this IS the expected "before" baseline showing zero tree-shaking.

### Step 2: Create benchmarks directory structure

Create `benchmarks/` with `README.md`, `.gitignore`, and the `apps/` subdirectories.

### Step 3: Create node-cli app

- `package.json` with `@alephium/web3` as a `file:` dependency pointing to the packed tarball
- `index.js` вЂ” CJS script that requires `isValidAddress` and validates a hardcoded address (`1DrDyTr9RpRsQnDnXo2YRiPzPW4ooHX5LLoqXrqfMrpQH`)

### Step 4: Create website app (Vite vanilla)

- `package.json` with Vite + `vite-plugin-node-polyfills` (required because current SDK uses Node builtins)
- `vite.config.js` with the polyfills plugin
- `index.html` + `src/main.js` вЂ” text input that validates addresses on keystrokes
- The polyfill requirement itself is a "before" data point

### Step 5: Create webapp app (Vite + React)

- Same as website but with React, `@vitejs/plugin-react`, TypeScript
- `App.tsx` with useState for the address input
- Also needs `vite-plugin-node-polyfills`

### Step 6: Create expo app

- Minimal Expo project with `expo`, `react`, `react-native`
- `metro.config.js` with `extraNodeModules` mapping for `crypto`, `stream`, `path`
- Additional dependencies: `crypto-browserify`, `stream-browserify`, `path-browserify`
- Build may fail вЂ” that failure IS a valid baseline data point to document

### Step 7: Create run.sh

The runner script will:
1. Build `@alephium/web3` and run `npm pack` to create a tarball
2. Run `size-limit` against the web3 package
3. Run `publint` and `@arethetypeswrong/cli` against the package
4. Collect dependency stats (direct dep count, UMD bundle size raw+gzip)
5. For each app: install deps, build, measure output sizes
6. Print a summary and save results to `benchmarks/results/`

### Step 8: Update ROADMAP.md

Add a new **Phase 0: Benchmarking Baseline** before Phase 1, describing:
- size-limit configuration for tracking import costs
- publint + @arethetypeswrong/cli for packaging health reports
- Four benchmark apps (node-cli, website, webapp, expo) that each import `isValidAddress`
- The runner script that captures all metrics
- Note that the same benchmarks will be re-run after each phase to track progress

## Key Design Decisions

- **Tarball-based linking**: Apps use `file:` pointing to the `npm pack` tarball rather than the source directory. This faithfully replicates what consumers get from npm.
- **Polyfill plugins in Vite apps**: The current SDK requires `vite-plugin-node-polyfills` for browser builds. Documenting this requirement (and its removal after modernization) is part of the benchmark story.
- **Expo may fail**: That's expected and valuable вЂ” it documents that the current SDK doesn't work in React Native without significant polyfill hacking.
